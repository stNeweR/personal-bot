# Clean Architecture Code Review

## Режимы работы

### 1. Проверка директории
Когда пользователь указывает директорию для проверки:
- Просканируйте структуру проекта в указанной директории
- Определите архитектурные слои (Entities, Use Cases, Interface Adapters, Frameworks)
- Проанализируйте зависимости между слоями и модулями
- Выявите нарушения правил Clean Architecture на уровне проекта

### 2. Проверка конкретного файла с зависимостями
Когда пользователь указывает конкретный файл для проверки:
- Прочитайте указанный файл
- Определите все импорты/зависимости этого файла
- Для каждого импорта локального файла — прочитайте и его тоже (рекурсивно, до 5 уровней глубины)
- Постройте граф зависимостей и проанализируйте:
  - Направление зависимостей (соответствует ли правилу "зависимости к центру")
  - Отсутствие циклических зависимостей
  - Правильное использование абстракций (интерфейсов) между слоями
  - Нарушение границ слоев

## Принципы Clean Architecture для проверки

### 1. Независимость от фреймворков

**Что проверять:**
- Бизнес-логика не зависит от внешних фреймворков
- Фреймворки используются только во внешних слоях
- Можно заменить фреймворк без изменения бизнес-логики

**Признаки нарушения:**
- Импорт фреймворков (Laravel, Symfony, Spring) в доменных сущностях
- Аннотации/атрибуты фреймворков в entity-классах
- Вызов статических методов фреймворка в бизнес-логике

**Как исправлять:**
- Переместите зависимость от фреймворка во внешний слой
- Используйте адаптеры для интеграции с фреймворком
- Создайте абстракцию над функциональностью фреймворка

### 2. Управляемые зависимости (Dependency Rule)

**Что проверять:**
- Зависимости направлены только внутрь (к центру архитектуры)
- Внутренние слои не знают о внешних слоях
- Внешние слои зависят от внутренних

**Слои (от центра к периферии):**
1. **Entities** (сущности) — бизнес-правила, самые важные объекты
2. **Use Cases** (сценарии использования) — бизнес-логика приложения
3. **Interface Adapters** (адаптеры интерфейсов) — контроллеры, презентеры, DTO
4. **Frameworks & Drivers** (фреймворки и драйверы) — БД, UI, внешние API

**Признаки нарушения:**
- Entity импортирует классы из Use Cases или Controllers
- Use Case импортирует контроллеры или модели БД
- Зависимость от конкретных реализаций БД во внутренних слоях

**Как исправлять:**
- Инвертируйте зависимость через интерфейсы
- Определите интерфейс во внутреннем слое, реализуйте во внешнем
- Используйте dependency injection

### 3. Организация по слоям

**Что проверять:**
- Четкое разделение на слои архитектуры
- Каждый слой имеет свою ответственность
- Слои изолированы друг от друга

**Ответственности слоев:**

| Слой | Ответственность | Примеры |
|------|----------------|---------|
| Entities | Бизнес-правила, доменные объекты | User, Order, Product |
| Use Cases | Бизнес-сценарии, координация | CreateUser, ProcessOrder |
| Interface Adapters | Преобразование данных, MVC | Controllers, DTOs, Presenters |
| Frameworks | Инфраструктура, внешние зависимости | Repositories, External APIs |

**Признаки нарушения:**
- Бизнес-логика в контроллерах
- SQL-запросы в use cases
- HTML/JSON формирование в entity

**Как исправлять:**
- Переместите код в соответствующий слой
- Используйте DTO для передачи данных между слоями
- Выделите логику в use cases

### 4. Входные и выходные порты (Ports & Adapters)

**Что проверять:**
- Порты четко определены (интерфейсы)
- Адаптеры реализуют порты
- Взаимодействие между слоями через порты

**Входные порты (Driving):**
- Определяют, как внешний мир взаимодействует с приложением
- Реализуются use cases
- Вызываются контроллерами

**Выходные порты (Driven):**
- Определяют, как приложение взаимодействует с внешним миром
- Определяются во внутреннем слое (use cases/entities)
- Реализуются во внешнем слое (repositories, external services)

**Признаки нарушения:**
- Прямой вызов repository из контроллера (минуя use case)
- Use case зависит от конкретной реализации repository
- Отсутствие интерфейсов для внешних зависимостей

**Как исправлять:**
- Определите интерфейс repository в слое use cases
- Реализуйте интерфейс в инфраструктурном слое
- Внедрите зависимость через конструктор use case

## Архитектурные паттерны для проверки

### Repository Pattern

**Что проверять:**
- [ ] Данные доступны через абстрактные интерфейсы
- [ ] Логика доступа к данным изолирована в repository
- [ ] Repository определено в доменном слое, реализовано в инфраструктурном
- [ ] Возвращаются доменные объекты, а не ORM-сущности

### Service Layer / Use Cases

**Что проверять:**
- [ ] Бизнес-логика сосредоточена в сервисах/use cases
- [ ] Сервисы не зависят от UI или конкретной персистентности
- [ ] Каждый use case соответствует одному сценарию использования
- [ ] Use cases координируют flow, а не содержат бизнес-правила

### DTO (Data Transfer Objects)

**Что проверять:**
- [ ] Используются для передачи данных между слоями
- [ ] Не содержат бизнес-логики
- [ ] Иммутабельны (желательно)
- [ ] Специфичны для каждого use case (не универсальные)

### Boundary / Interface Objects

**Что проверять:**
- [ ] Входные данные преобразуются в DTO на границе
- [ ] Выходные данные преобразуются из domain-объектов на границе
- [ ] Контроллеры тонкие, только преобразование и делегирование

## Проверка архитектурных ограничений

### Направление зависимостей

**Чеклист:**
- [ ] Зависимости всегда направлены к центру архитектуры
- [ ] Внутренние слои не импортируют внешние
- [ ] Внешние слои реализуют интерфейсы, определенные внутри

### Циклические зависимости

**Что проверять:**
- Отсутствуют циклы между модулями
- Отсутствуют циклы между классами внутри слоя
- Граф зависимостей ациклический

### Границы контекстов (Bounded Contexts)

**Что проверять:**
- Разные контексты изолированы
- Нет прямой зависимости между контекстами
- Используется ACL (Anti-Corruption Layer) при интеграции

## Критерии оценки

### Структура проекта
- [ ] Четкое разделение на директории по слоям
- [ ] Имена директорий отражают архитектурную роль
- [ ] Модули слабо связаны

### Зависимости
- [ ] Нет импортов из внешних слоев во внутренние
- [ ] Интерфейсы определены в правильном слое
- [ ] Dependency injection используется корректно

### Изоляция
- [ ] Бизнес-логика тестируется без фреймворков
- [ ] Use cases не зависят от БД напрямую
- [ ] Контроллеры не содержат бизнес-логики

## Формат вывода результатов

### При проверке директории:

1. **Карта архитектуры**:
   - Обнаруженные слои и их расположение
   - Основные модули/контексты

2. **Нарушения Dependency Rule**:
   - Список импортов, нарушающих направление зависимостей
   - От какого слоя к какому идет нарушение

3. **Проблемы организации слоев**:
   - Бизнес-логика не в том слое
   - Смешение ответственностей

4. **Циклические зависимости**:
   - Обнаруженные циклы в графе зависимостей

5. **Рекомендации**:
   - Приоритизированный список изменений
   - Примеры рефакторинга

### При проверке файла с зависимостями:

1. **Граф зависимостей**:
   ```
   проверяемый_файл.php
   ├── зависимость_1.php
   │   └── подзависимость_1_1.php
   ├── зависимость_2.php
   └── зависимость_3.php
   ```

2. **Анализ каждого файла**:
   - Слой, к которому относится файл
   - Нарушения Clean Architecture в файле
   - Зависимости, которые нарушают правила

3. **Анализ направлений зависимостей**:
   - Какие зависимости корректны (направлены к центру)
   - Какие зависимости нарушают правило

4. **Общий вердикт**:
   - Оценка соответствия Clean Architecture
   - Критические нарушения
   - Рекомендации по рефакторингу

## Примеры рекомендаций

**Плохо:**
```
Нарушена Clean Architecture. Переместите код в правильный слой.
```

**Хорошо:**
```
Нарушение Dependency Rule:
- Файл: src/Domain/User/User.php (слой: Entities)
- Нарушение: импортирует src/Infrastructure/Database/ORM/Entity.php
- Проблема: Entity-слой зависит от инфраструктуры

Рекомендация:
1. Уберите зависимость от ORM в доменном слое
2. Создайте чистый PHP-класс для доменной сущности
3. Реализуйте маппинг между ORM и доменной сущностью в инфраструктурном слое

Пример доменной сущности:
class User {
    public function __construct(
        private UserId $id,
        private Email $email,
        private Name $name
    ) {}
    
    // Бизнес-логика здесь, без ORM-аннотаций
}
```
