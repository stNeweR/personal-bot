# SOLID Code Review

## Режимы работы

### 1. Проверка директории
Когда пользователь указывает директорию для проверки:
- Просканируйте все файлы кода в указанной директории и поддиректориях
- Проанализируйте каждый файл на соответствие принципам SOLID
- Выявите общие проблемы на уровне модуля/пакета
- Сгруппируйте нарушения по файлам и принципам

### 2. Проверка конкретного файла с зависимостями
Когда пользователь указывает конкретный файл для проверки:
- Прочитайте указанный файл
- Определите все импорты/зависимости этого файла
- Для каждого импорта локального файла — прочитайте и его тоже (рекурсивно, до 3 уровней глубины)
- Постройте граф зависимостей и проанализируйте код всех файлов на соответствие SOLID

## Принципы SOLID для проверки

### Single Responsibility Principle (Принцип единственной ответственности)

**Что проверять:**
- Каждый класс имеет только одну причину для изменения
- Методы выполняют одну логическую задачу
- Классы не делают слишком много (признаки "God Class")

**Признаки нарушения:**
- Класс содержит более 7-10 публичных методов
- Метод длиннее 20-30 строк
- В имени класса есть "and", "or" (например, `UserManagerAndLogger`)
- Метод оперирует данными из разных доменов

**Как исправлять:**
- Выделите отдельные классы для разных ответственностей
- Разбейте большие методы на меньшие с понятными именами
- Используйте композицию вместо наследования

### Open/Closed Principle (Принцип открытости/закрытости)

**Что проверять:**
- Код открыт для расширения, но закрыт для модификации
- Используются абстракции и интерфейсы для расширения
- Новая функциональность добавляется через новые классы, а не изменение существующих

**Признаки нарушения:**
- Большие switch/if-else конструкции с проверкой типов
- Изменение существующего кода при добавлении новой функциональности
- Отсутствие полиморфизма там, где он уместен

**Как исправлять:**
- Используйте стратегию, фабрику или другие паттерны для устранения условной логики
- Выделите интерфейс и создавайте новые реализации вместо модификации
- Применяйте dependency injection

### Liskov Substitution Principle (Принцип подстановки Лисков)

**Что проверять:**
- Объекты заменяемы экземплярами их подтипов без изменения правильности программы
- Наследование реализовано корректно
- Производные классы действительно являются подтипами базового

**Признаки нарушения:**
- Дочерний класс выбрасывает исключения, которые не ожидаются от базового
- Дочерний класс переопределяет метод с более строгими предусловиями
- Проверки типа instanceof перед использованием базового класса
- Методы возвращают null или специальные значения в подклассах

**Как исправлять:**
- Пересмотрите иерархию наследования
- Используйте композицию вместо наследования
- Убедитесь, что контракты базового класса соблюдаются

### Interface Segregation Principle (Принцип разделения интерфейса)

**Что проверять:**
- Клиенты не зависят от интерфейсов, которые они не используют
- Интерфейсы специфичны и сфокусированы
- Интерфейсы разбиты на мелкие специализированные части

**Признаки нарушения:**
- "Жирные" интерфейсы с множеством методов
- Классы реализуют методы, которые возвращают NotImplementedException
- Классы вынуждены зависеть от методов, которые они не используют

**Как исправлять:**
- Разбейте большие интерфейсы на специализированные
- Создайте отдельные интерфейсы для разных клиентов
- Используйте адаптеры для совместимости

### Dependency Inversion Principle (Принцип инверсии зависимостей)

**Что проверять:**
- Модули верхних уровней не зависят от модулей нижних уровней
- Оба зависят от абстракций
- Зависимости направлены к абстракциям, а не конкретным реализациям

**Признаки нарушения:**
- Конкретные классы создаются внутри других классов (new ConcreteClass())
- Зависимости от конкретных реализаций вместо интерфейсов
- Отсутствие dependency injection

**Как исправлять:**
- Внедрите зависимости через конструктор или методы
- Зависите от интерфейсов/абстрактных классов
- Используйте DI-контейнер или фабрику

## Критерии оценки

### Структурные
- [ ] Классы имеют одну ответственность
- [ ] Методы не длиннее 20-30 строк
- [ ] Файлы не больше 500 строк
- [ ] Цикломатическая сложность методов ≤ 10

### Именование
- [ ] Имена отражают назначение
- [ ] Единый стиль именования
- [ ] Нет неясных сокращений

### Зависимости
- [ ] Используются интерфейсы для абстракции
- [ ] Dependency injection реализован
- [ ] Нет циклических зависимостей

### Тестируемость
- [ ] Код легко тестируется изолированно
- [ ] Зависимости можно замокать

## Формат вывода результатов

### При проверке директории:
1. **Обзор**: количество проверенных файлов, общий уровень соответствия SOLID
2. **Нарушения по принципам**: сгруппируйте проблемы по каждому из 5 принципов
3. **Критические проблемы**: файлы с наибольшим количеством нарушений
4. **Рекомендации**: приоритизированный список рефакторинга

### При проверке файла с зависимостями:
1. **Граф зависимостей**: список всех проверенных файлов
2. **Анализ по файлам**: нарушения SOLID в каждом файле
3. **Детализация по принципам**: для каждого нарушения укажите:
   - Какой принцип нарушен
   - Строка/метод/класс
   - Почему это нарушение
   - Как исправить (с примером кода)
4. **Общий вердикт**: оценка соответствия SOLID

## Примеры рекомендаций

**Плохо:**
```
Нарушен SRP. Разбейте класс.
```

**Хорошо:**
```
Нарушен Single Responsibility Principle:
- Класс UserService управляет пользователями И отправляет email (строки 45-67)
- Метод register() содержит логику валидации, сохранения и нотификации

Рекомендация:
1. Выделите валидацию в отдельный класс UserValidator
2. Выделите отправку email в EmailNotifier
3. UserService должен координировать, а не выполнять

Пример:
class UserService {
    constructor(
        private UserValidator $validator,
        private UserRepository $repo,
        private EmailNotifier $notifier
    ) {}
    
    public function register(UserDto $dto): User {
        $this->validator->validate($dto);
        $user = $this->repo->save($dto);
        $this->notifier->sendWelcome($user);
        return $user;
    }
}
```
